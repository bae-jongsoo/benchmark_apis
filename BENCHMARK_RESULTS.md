- 날짜: 2026-04-02
- 동시 사용자: 1,000 VUs (ramp-up 10s + steady 30s)

## Phase 3 — Fake IO (5ms sleep + JSON 직렬화)

짧은 IO 대기 + 직렬화. 동시성과 언어 성능이 둘 다 영향을 주는 구간.

| 구성 | RPS | P50(ms) | P95(ms) | P99(ms) | CPU(%) | Mem(MB) | RPS/CPU% |
|---|---:|---:|---:|---:|---:|---:|---:|
| spring-default | 37,097 | 26.1 | 31.3 | 45.0 | 212.4 | 394.9 | **174.7** |
| fastapi-4w | 12,787 | 78.6 | 97.9 | 107.9 | 405.6 | 264.9 | 31.5 |
| fastapi-2w | 7,133 | 140.3 | 176.3 | 184.3 | 191.3 | 158.3 | 37.3 |
| django-16w | 2,887 | 334.2 | 349.8 | 356.9 | 110.0 | 445.1 | 26.2 |
| spring-4t | 702 | 1,417.8 | 1,428.9 | 1,430.6 | 24.8 | 267.4 | 28.3 |

> `spring-default`(200 스레드)가 CPU 212%로 37,097 RPS. Jackson이 Pydantic 대비 CPU 효율 약 5배. 다만 동기 4스레드로 제한하면(`spring-4t` 702 vs `django-4w` 668) 언어 차이가 거의 사라진다.

## 정리

### CPU 연산이 많으면 Java가 확실히 빠르다

JIT 컴파일 vs 인터프리터 차이가 고스란히 나온다. CPU 집약적 워크로드에서는 언어 선택이 곧 성능.

### IO 대기가 길어지면 언어 성능은 상관없다

100ms IO 앞에서는 Java든 Python이든 놀고 있는 시간이 99%. 이때는 대기 중에 다른 요청을 얼마나 처리할 수 있느냐(동시성 모델)가 전부.

### 동기 모델은 스레드/워커 수가 곧 천장

`spring-4t`과 `django-4w`가 IO-bound에서 38 RPS로 동일. Fake IO에서도 702 vs 668. 동기 모델에서는 언어가 아무리 빨라도 동시 처리 단위 수를 넘지 못한다.

### 그래서 뭘 쓸까

| 워크로드 | 선택지 | 이유 |
|---|---|---|
| CPU 집약 (영상처리, 연산 등) | Spring (Java) | 연산 성능 차이가 크다 |
| IO 위주 일반 웹서비스 | FastAPI (Python) | 성능 충분하고 코드가 짧다 |
| 대규모 동시접속 | FastAPI 또는 Spring WebFlux | async/reactive 필요 |

보통 웹 서비스는 요청 시간의 대부분이 DB 쿼리, 외부 API 호출 같은 IO 대기다. 그래서 언어 성능보다는 동시성 모델과 개발 생산성이 더 현실적인 선택 기준이 된다.
